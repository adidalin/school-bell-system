# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

> ⬜ 未开始

## 五、预约铃声（铃音临时覆盖）

### 专家讨论

**张老师（产品）**：

> 这个需求很实在。课表是固定的框架，但实际教学中有太多临时变动——下周要考试了，上课铃想换成"考生入场"提示音；或者这一周搞活动，下课铃改成轻音乐。老师不可能为了这个去改整个课表。
>
> 核心逻辑就是一句话：**打铃前先查有没有预约，有就用预约的，没有就用课表默认的。**
>
> 使用场景：
> - 下周一到周五，把上午第二节下课铃换成指定铃声
> - 仅今天，把全校的上课铃换成消防演练提示音
> - 下个月整个月，把初中部的放学铃换成校歌

**陈工（架构）**：

> 可行，改动集中在数据层和调度层，不涉及音频引擎。
>
> **数据模型**：新建一张 `bell_overrides` 表
>
> ```sql
> CREATE TABLE bell_overrides (
>     id          INTEGER PRIMARY KEY AUTOINCREMENT,
>     zone_id     TEXT NOT NULL,       -- 区域（为空表示全部区域）
>     bell_type   TEXT NOT NULL,       -- 铃音类型（class_start / class_end / ...）
>     time        TEXT,                -- 具体时间 HH:MM（为空表示全天）
>     start_date  TEXT NOT NULL,       -- 生效日期起
>     end_date    TEXT NOT NULL,       -- 生效日期止
>     bell_id     INTEGER NOT NULL,    -- 替换为哪个铃声
>     days        TEXT,                -- 适用星期（1-7逗号分隔，NULL=每天）
>     priority    INTEGER DEFAULT 0,   -- 优先级（数字越大越优先）
>     desc        TEXT,                -- 备注
>     created_at  TEXT DEFAULT (datetime('now'))
> );
> ```
>
> **调度逻辑改动**（关键点，仅改 `engine.py` 中的 bell_check_task）：
>
> ```
> 当前: task.bell_type → 查铃声 → 播放
> 改后: task.bell_type → 查 bell_overrides（按 zone+type+time+日期 匹配）
>                       → 有匹配? 用 override.bell_id : 用 task 默认 bell_type → 播放
> ```
>
> 匹配规则：
> 1. 精确匹配：zone_id + bell_type + time + 日期包含今天 + days 包含今天星期
> 2. 范围匹配：zone_id + bell_type + time 为空（全天）+ 日期范围
> 3. 通配匹配：zone_id 为空（所有区域）+ bell_type
> 4. 如果有多个匹配，取 priority 最高的
>
> **API**（需新增）：
> - `GET /api/overrides` — 查预约列表（支持 zone_id、date 过滤）
> - `POST /api/overrides` — 新建预约
> - `PUT /api/overrides/:id` — 修改
> - `DELETE /api/overrides/:id` — 删除
>
> **不改的部分**：
> - `engine.py` 音频播放逻辑不动
> - `engine.py` 的 `play()` 方法不动
> - 课表 CRUD 不动
> - DB 现有表不动

**王工（运维）**：

> 注意一件事：预约铃声有"有效期"，到期要自动清理或标记过期。如果预约了一大堆不清理，每次打铃都扫描一遍，效率会下降。
>
> 建议：
> - 每次 scheduler 启动时（或每天凌晨）自动删除 `end_date < today` 的记录
> - `days` 字段用逗号分隔的星期数字，和现有 task 的 `days` 字段保持一致
> - 前端给个"预约日历"视图，一目了然哪天哪节课被覆盖了

**小花荣（用户）**：

> 我问个实际的问题——如果我先预约了下周用铃声A，过了两天我又想改回默认了，是不是要把预约删掉？
>
> **陈工**：对。删除预约 = 恢复默认。也可以在 UI 上加个"恢复默认"按钮，其实就是 DELETE 操作。
>
> **小花荣**：那如果预约和课表默认是同一个铃声呢？
>
> **陈工**：系统在打铃时会先判断——如果预约的 bell_id 和课表默认的 bell_id 相同，就跳过预约逻辑，不产生额外查询。用户根本感觉不到。
