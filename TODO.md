# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

- **有一个独立的页面叫"预约铃声"**，不是嵌入课表
- 页面加载课表数据，展示所有区域下的任务列表（作为选择来源）

## 六、数据持久化分析（断电重启后数据是否保持）

### 问题

用户问：**课表数据、预约铃声数据，断电重启后是否还在？**

### 专家讨论

**陈工（架构）**：

> 回答：**全部保持。** 现有系统全部数据都存放在 `data/school_bell.db`，这是一个 SQLite 数据库文件，存储在硬盘上。断电不丢失数据。
>
> 我来分三层说清楚：

#### 第一层：存储层（SQLite）

```
数据存放位置：
  C:\Users\server\Desktop\钢城智慧铃声系统\data\school_bell.db  ✓ 硬盘文件
```

现有数据全部在 SQLite 中：

| 数据 | 表名 | 是否持久 |
|------|------|---------|
| 区域配置（名称、音频设备、音量） | `zones` | ✅ 硬盘 |
| 课表方案 | `schedules` | ✅ 硬盘 |
| 课表任务（时间、铃型、星期） | `tasks` | ✅ 硬盘 |
| 铃声库（文件名、类型） | `bells` | ✅ 硬盘 |
| 今日例外、明日覆盖 | `today_exceptions`, `tomorrow_overrides` | ✅ 硬盘 |
| 节假日 | `holidays` | ✅ 硬盘 |
| 运行日志 | `logs` | ✅ 硬盘 |
| **新建：预约铃声** | `task_overrides` | ✅ **硬盘** |

SQLite 使用 WAL（Write-Ahead Logging）模式，具备**崩溃安全**特性。写入时先写 WAL 再写 DB，断电恢复时自动回放 WAL 或回滚未完成的事务。不会出现"写了一半断电导致数据损坏"的情况。

> **一句话：只要点过"保存"，数据就已经在硬盘上了。拔电源再开机，数据还在。**

#### 第二层：调度层（APScheduler）

APScheduler 是内存调度器，断电后**调度状态会丢失**，但它会在启动时重新从数据库读取课表任务重建调度。

重启时序：

```
① 系统开机
② run.py 启动
③ init_system() 执行
   ├── init_db() → 打开 school_bell.db ✓
   ├── _load_zone_devices() → 读 zones 表 → zone_devices ✓
   └── scheduler.start() → APScheduler 启动
④ bell_check_task 开始每秒执行一次
   └── 读 tasks 表 → 匹配当前时间 → 打铃
```

**重启后确实会丢掉一些信息：**

| 会丢的 | 原因 | 影响 |
|--------|------|------|
| 正在播放中的铃声 | 铃声在内存中播放，断电即停 | 重新开机后不会继续播 |
| 已执行过的打铃记录 | `logs` 表持久，但本次 session 的日志缓冲区可能没刷盘 | 差最后几秒的日志 |
| 定时器"剩余等待时间" | APScheduler 下次触发时间是内存中的 | 重启后重新计算 |

**不会丢的：**

| 不会丢的 | 原因 |
|----------|------|
| 课表任务（含新增、修改） | 全在 SQLite |
| 区域配置 | 全在 SQLite |
| 铃声文件（WAV/MP3） | 全在硬盘 `static/sounds/` |
| **预约铃声记录** | 全在 SQLite `task_overrides` 表 |
| 今日例外、明日覆盖 | 全在 SQLite |

**边界情况——停电期间错过的铃声怎么办？**

假设停电发生在 08:00 - 08:30：

| 时间 | 事件 | 开机后 |
|------|------|--------|
| 07:55 | 正常打铃 ✅ | — |
| 08:00 | ⚡断电，第2节上课铃没响 | 08:31 恢复供电，系统启动 |
| 08:05-08:30 | 停电中，所有铃都没响 | 不补打（已错过） |
| 08:31 | 系统启动 | 判断当前时间 08:31，无匹配任务 → 等待下一节课 |
| 08:50 | 第3节上课铃 | ✅ 正常响起 |

> 结论：**错过的铃不补打**，之后恢复正常。学校场景下这是可接受的——电都停了，打了也没人听。

#### 第三层：预约铃声持久化（新建功能）

`task_overrides` 表和现有表完全一样存放在 SQLite 中：

```sql
CREATE TABLE task_overrides (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    INTEGER NOT NULL,
    start_date TEXT NOT NULL,
    end_date   TEXT NOT NULL,
    bell_id    INTEGER NOT NULL,
    desc       TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
```

新建预约时：
```
POST /api/overrides → INSERT INTO task_overrides → conn.commit() → 落盘 ✓
```

重启后：
```
bell_check_task → SELECT * FROM task_overrides WHERE task_id=? AND ? BETWEEN start_date AND end_date
                → 能查到之前建的记录 ✓ → 正常使用预约铃
```

**预约铃声和普通课表任务在持久化上没有任何区别。**

**王工（运维）**：

> 我补充一个运维层面的建议：
>
> 1. **数据库备份**：每天凌晨自动把 `school_bell.db` 复制到 `backups/` 目录，保留最近 7 天
> 2. **启动日志**：每次启动在日志里记一条 `System started after power loss`，方便排查
> 3. **UPS 不间断电源**：如果学校条件允许，给主机配一个 UPS，断电后还能坚持 10-15 分钟正常打铃
>
> 这些都已写入"四、其他待优化项"中。

**张老师总结**：

> 用户问的核心问题就是"我辛辛苦苦配的课表、建的预约，断电还有没有"——答案是 **有，全在硬盘里。** 拔电重启跟关掉浏览器再打开一样，数据丝毫无损。放心用。

### 关键定论

| 问题 | 回答 |
|------|------|
| 课表数据断电会丢吗？ | ❌ 不会。全部在 SQLite 硬盘文件 |
| 预约铃声数据断电会丢吗？ | ❌ 不会。和课表同一张 DB |

| 是否需要 UPS？ | 可选。不是必须，但有更好（能覆盖短暂停电） |

### 技术方案选择：edge-tts 详解

**为什么选 edge-tts 而不是微软 SAPI5 或 pyttsx3：**

| 对比项 | edge-tts（Edge 在线 TTS） | pyttsx3（Windows SAPI5） |
|--------|--------------------------|-------------------------|
| **GitHub Stars** | 11.3k | 2.5k |
| **中文语音质量** | ⭐⭐⭐⭐⭐ 神经网络语音，自然流畅 | ⭐⭐⭐ 较机械，微软旧引擎 |
| **中文可选语音** | 10+（Xiaoxiao/Yunxi/Yunyang/Xiaohan...） | 1-2（取决于 Windows 版本） |
| **语音风格** | 支持（general/cheerful/sad/call等） | 不支持 |
| **语速/音量/音调** | ✅ 全部支持 | ✅ 全部支持 |
| **导出格式** | MP3 / raw PCM | WAV |
| **需要网络** | ✅ 是（调用微软云服务） | ❌ 否（完全离线） |
| **需要 Edge 浏览器** | ❌ 不需要 | — |
| **需要 API Key** | ❌ 完全免费 | — |
| **跨平台** | ✅ Linux/macOS/Windows | ✅ Linux/macOS/Windows |

**核心关注点——断网怎么办：**

> edge-tts 需要互联网连接才能生成语音。但铃声编辑的使用场景是"管理员在电脑前操作"，每次只生成一条短语音（5-30 秒），网络中断的可能性很低。即使偶尔断网，可以提示"当前无网络，TTS 功能不可用"。
>
> **更稳妥的方案：双引擎策略**
> ```
> def generate_tts(text, voice="zh-CN-XiaoxiaoNeural"):
>     try:
>         # 首选 edge-tts（高质量）
>         return edge_tts_generate(text, voice)
>     except (ConnectionError, TimeoutError):
>         # 降级到 pyttsx3（离线）
>         return pyttsx3_generate(text, "Chinese")
> ```
> 
> 一旦 TTS 生成完毕保存为 WAV 文件，之后播放完全不依赖网络。**网络只影响"创建铃声"这一刻，不影响"响铃"这一刻。**

#### 音频格式问题的解决方案

edge-tts 默认输出 MP3，而我们的系统播放 WAV。但 edge-tts 支持 `raw-24khz-16bit-mono-pcm` 输出格式——以原始 PCM 流替代 MP3，在 Python 端加上 WAV 文件头即可转成标准 WAV：

```
edge-tts → 设置 output_format="raw-24khz-16bit-mono-pcm"
         → 收到原始 PCM bytes（16-bit little-endian，24kHz，单声道）
         → Python 用 wave 模块加上 RIFF 头
         → 保存为 .wav 文件（零 ffmpeg，零额外依赖）
```

Python 代码示意（3 行）：
```python
import wave, struct
with wave.open("output.wav", "wb") as w:
    w.setnchannels(1); w.setsampwidth(2); w.setframerate(24000)
    w.writeframes(pcm_bytes)
```

#### 对现有系统的兼容性分析

**3 个关键保证：**

> 1. ❌ **不修改现有打铃逻辑。** 铃声编辑是独立功能，新生成的铃声只是 `bells` 表多一条记录 + `static/sounds/` 多一个 WAV 文件。打铃路径 `task → bell_type → bell.filepath` 不变。
>
> 2. ❌ **不引入新运行时依赖。** `edge-tts` 只在呼叫 `/api/bell-editor/tts` 时才被加载。打铃线程和调度线程完全不受影响。`pip install edge-tts` 后导入 `edge_tts` 本身不阻塞，不抢占 GIL。
>
> 3. ❌ **不修改现有数据库表。** 铃声编辑只操作 `bells` 表（INSERT）。现有 `tasks`, `schedules`, `zones`, `task_overrides` 全部不动。

**专家讨论：**

**陈工（架构）**：

> edge-tts 的架构决定了它和我们的系统是一个松耦合关系——它只在"创建铃声"时才被调用，打完电话就挂断。两个独立进程的感觉，不会互相影响。
>
> 唯一要注意的是：
>
> 1. **异步调用**。edge-tts 使用 `asyncio`（aiohttp WebSocket），而 Flask 是同步的。需要 `asyncio.run()` 包裹，或者 `loop.run_until_complete()`。Flask 路由里面直接 `asyncio.run(communicate(...))` 即可——每次调用创建一个新 event loop，用完就丢，不会和 APScheduler 的 event loop 冲突（APScheduler 默认不使用 asyncio）。
>
> 2. **超时处理**。网络请求可能超时。设置 15 秒超时时间，超时后回到降级策略或返回错误信息给前端。
>
> 3. **临时文件清理**。TTS 生成的临时文件要及时清理。建议全程在内存中处理（PCM bytes → numpy array → 混音 → 写出最终文件），不写中间文件。
>
> 4. **edge-tts 是否会被微软封禁？** 这是一个社区维护的项目，通过模拟 Edge 浏览器的 WebSocket 握手来访问公开服务。微软没有官方禁止，但也没有保证永远可用。这个项目已经存在 3 年+，Star 11k+，社区活跃。如果真的被封禁，我们可以切到 pyttsx3，或者当时再评估其他方案。

**张老师（产品）**：

> 用户要的是两件事：
>
> 1. 铃声编辑功能"能不能做"——✅ 完全可以
> 2. 稳定的 TTS 语音——⏳ 需要用真实环境测试
>
> 我的建议是这样：
>
> - 先用 edge-tts 开发完成
> - 部署后在现场测试 TTS 效果（需要网络）
> - 如果网络不稳定，后台加装 pyttsx3 作为 fallback
>
> 学校的网络一般不会太差，至少办公室的电脑肯定是联网的。管理员在电脑前操作铃声编辑，网络十拿九稳。
>
> 流程设计上，让用户操作的前两步（导入音频、裁剪）完全离线可用，只有 TTS 这一步需要网络。这样即使断网，用户也能完成其他编辑工作。

**王工（运维）**：

> 我说一下运维角度的风险评估：
>
> **风险矩阵：**
>
> | 风险 | 概率 | 影响 | 应对 |
> |------|------|------|------|
> | 编辑时断网 | 低 | TTS 失败，不能生成语音 | fallback 到 pyttsx3 + 前端提示"网络不可用，使用离线语音" |
> | edge-tts 被微软封禁 | 极低 | 整个 TTS 功能不可用 | 切到 pyttsx3，或等社区修复 |
> | 生成的铃声文件过多 | 低 | 硬盘空间占用 | 限制最多生成 200 个，超限提示清理 |
> | 用户把乱七八糟的音频导入 | 中 | 音质差、尺度过大 | 前端限制文件大小 10MB、格式 WAV/MP3 |
>
> **关于离线降级方案 pyttsx3：**
>
> `pip install pyttsx3` 本身不含任何安全隐患。它只是调用 Windows 系统的 SAPI5 COM 接口，和系统自带的语音功能一样可靠。即使 asyncio 失败，它也不会影响 Flask 的主线程——降级的 if 分支在 `except` 块里，互斥。
>
> **结论：零风险，不会引入新 bug。**
