# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

## 五、预约铃声（独立页面，绑定 task 的临时换铃）

- **有一个独立的页面叫"预约铃声"**，不是嵌入课表
- 页面加载课表数据，展示所有区域下的任务列表（作为选择来源）

## 六、数据持久化分析（断电重启后数据是否保持）

### 问题

用户问：**课表数据、预约铃声数据，断电重启后是否还在？**

### 专家讨论

**陈工（架构）**：

> 回答：**全部保持。** 现有系统全部数据都存放在 `data/school_bell.db`，这是一个 SQLite 数据库文件，存储在硬盘上。断电不丢失数据。
>
> 我来分三层说清楚：

#### 第一层：存储层（SQLite）

```
数据存放位置：
  C:\Users\server\Desktop\钢城智慧铃声系统\data\school_bell.db  ✓ 硬盘文件
```

现有数据全部在 SQLite 中：

| 数据 | 表名 | 是否持久 |
|------|------|---------|
| 区域配置（名称、音频设备、音量） | `zones` | ✅ 硬盘 |
| 课表方案 | `schedules` | ✅ 硬盘 |
| 课表任务（时间、铃型、星期） | `tasks` | ✅ 硬盘 |
| 铃声库（文件名、类型） | `bells` | ✅ 硬盘 |
| 今日例外、明日覆盖 | `today_exceptions`, `tomorrow_overrides` | ✅ 硬盘 |
| 节假日 | `holidays` | ✅ 硬盘 |
| 运行日志 | `logs` | ✅ 硬盘 |
| **新建：预约铃声** | `task_overrides` | ✅ **硬盘** |

SQLite 使用 WAL（Write-Ahead Logging）模式，具备**崩溃安全**特性。写入时先写 WAL 再写 DB，断电恢复时自动回放 WAL 或回滚未完成的事务。不会出现"写了一半断电导致数据损坏"的情况。

> **一句话：只要点过"保存"，数据就已经在硬盘上了。拔电源再开机，数据还在。**

#### 第二层：调度层（APScheduler）

APScheduler 是内存调度器，断电后**调度状态会丢失**，但它会在启动时重新从数据库读取课表任务重建调度。

重启时序：

```
① 系统开机
② run.py 启动
③ init_system() 执行
   ├── init_db() → 打开 school_bell.db ✓
   ├── _load_zone_devices() → 读 zones 表 → zone_devices ✓
   └── scheduler.start() → APScheduler 启动
④ bell_check_task 开始每秒执行一次
   └── 读 tasks 表 → 匹配当前时间 → 打铃
```

**重启后确实会丢掉一些信息：**

| 会丢的 | 原因 | 影响 |
|--------|------|------|
| 正在播放中的铃声 | 铃声在内存中播放，断电即停 | 重新开机后不会继续播 |
| 已执行过的打铃记录 | `logs` 表持久，但本次 session 的日志缓冲区可能没刷盘 | 差最后几秒的日志 |
| 定时器"剩余等待时间" | APScheduler 下次触发时间是内存中的 | 重启后重新计算 |

**不会丢的：**

| 不会丢的 | 原因 |
|----------|------|
| 课表任务（含新增、修改） | 全在 SQLite |
| 区域配置 | 全在 SQLite |
| 铃声文件（WAV/MP3） | 全在硬盘 `static/sounds/` |
| **预约铃声记录** | 全在 SQLite `task_overrides` 表 |
| 今日例外、明日覆盖 | 全在 SQLite |

**边界情况——停电期间错过的铃声怎么办？**

假设停电发生在 08:00 - 08:30：

| 时间 | 事件 | 开机后 |
|------|------|--------|
| 07:55 | 正常打铃 ✅ | — |
| 08:00 | ⚡断电，第2节上课铃没响 | 08:31 恢复供电，系统启动 |
| 08:05-08:30 | 停电中，所有铃都没响 | 不补打（已错过） |
| 08:31 | 系统启动 | 判断当前时间 08:31，无匹配任务 → 等待下一节课 |
| 08:50 | 第3节上课铃 | ✅ 正常响起 |

> 结论：**错过的铃不补打**，之后恢复正常。学校场景下这是可接受的——电都停了，打了也没人听。

#### 第三层：预约铃声持久化（新建功能）

`task_overrides` 表和现有表完全一样存放在 SQLite 中：

```sql
CREATE TABLE task_overrides (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    INTEGER NOT NULL,
    start_date TEXT NOT NULL,
    end_date   TEXT NOT NULL,
    bell_id    INTEGER NOT NULL,
    desc       TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
```

新建预约时：
```
POST /api/overrides → INSERT INTO task_overrides → conn.commit() → 落盘 ✓
```

重启后：
```
bell_check_task → SELECT * FROM task_overrides WHERE task_id=? AND ? BETWEEN start_date AND end_date
                → 能查到之前建的记录 ✓ → 正常使用预约铃
```

**预约铃声和普通课表任务在持久化上没有任何区别。**

**王工（运维）**：

> 我补充一个运维层面的建议：
>
> 1. **数据库备份**：每天凌晨自动把 `school_bell.db` 复制到 `backups/` 目录，保留最近 7 天
> 2. **启动日志**：每次启动在日志里记一条 `System started after power loss`，方便排查
> 3. **UPS 不间断电源**：如果学校条件允许，给主机配一个 UPS，断电后还能坚持 10-15 分钟正常打铃
>
> 这些都已写入"四、其他待优化项"中。

**张老师总结**：

> 用户问的核心问题就是"我辛辛苦苦配的课表、建的预约，断电还有没有"——答案是 **有，全在硬盘里。** 拔电重启跟关掉浏览器再打开一样，数据丝毫无损。放心用。

### 关键定论

| 问题 | 回答 |
|------|------|
| 课表数据断电会丢吗？ | ❌ 不会。全部在 SQLite 硬盘文件 |
| 预约铃声数据断电会丢吗？ | ❌ 不会。和课表同一张 DB |
| 正在播放的铃断电会续吗？ | ❌ 不会续播（断电时中断即丢，开机后不补） |
| 停电期间错过的铃会补吗？ | ❌ 不补。恢复后从当前时间继续 |
| 数据会损坏吗？ | ✅ SQLite WAL 模式保证崩溃安全 |
| 是否需要 UPS？ | 可选。不是必须，但有更好（能覆盖短暂停电） |

> 1. **功能不全**。没有一个项目同时满足"TTS + 裁剪 + 混音 + 保存到服务器"四件套。AudioMass 有裁剪没 TTS，tts.rocks 有 TTS 没裁剪，Audio-Cutter-Web 有裁剪没后端保存。
>
> 2. **技术栈不匹配**。几乎全部是 npm/webpack/vite 构建的 SPA。我们的系统是 pure HTML + vanilla JS + Flask Jinja2 模板，没有任何构建工具。强行引入 npm 项目要先搭 webpack，为了一个编辑功能搞一套构建体系，得不偿失。
>
> 3. **没有"保存到服务器"的能力**。所有纯前端工具裁剪完只能下载到本地，不能 POST 到服务器写入 `bells` 表和 `static/sounds/` 目录。这是本质差异——我们的铃声需要服务端确认和持久化。
>
> **但是参考价值很大。** 建议策略：
>
> | 项目 | 参考什么 | 怎么参考 |
> |------|---------|---------|
> | **AudioMass** | Canvas 波形渲染 | `ui.js` 中的 draw 函数（纯 JS，零依赖） |
> | **AudioMass** | 鼠标选区拖拽 | `drag.js` 的事件处理逻辑 |
> | **web-audio-visualizer** | Flask API 设计 | 和我们的架构完全一致 |
> | **tts.rocks** | TTS 前端 UI 布局 | 语音选择、语速控制、预览播放 |
>
> **结论：参考不照搬。自己写一个轻量版本（~200行 JS + ~150 行 Python），比集成任何现有项目更省力、更可控。**

**张老师（产品）——"用户体验上能用吗？"**

> **AudioMass（2.8k⭐）——最接近，但操作链断裂**
>
> 如果集成 AudioMass，用户流程是：
> ```
> 打开 AudioMass → 导入音频 → 裁剪 → 下载 WAV → 回到铃声系统 → 上传 → 保存
> ```
> 操作在两个工具之间跳转，属于"集成但不好用"。但 AudioMass 的**波形渲染和选区拖拽的交互设计**是业界标杆，值得模仿。
>
> **tts.rocks（30⭐）——有趣的 TTS 前端**
>
> 它在浏览器端集成了 Kokoro/Piper/eSpeak 等 TTS 引擎，但：
> - AI 模型很大（~100MB+），首次加载慢
> - 中文效果不如 edge-tts 云端神经网络语音（Xiaoxiao/Yunxi）
> - 参考它的语音选择 UI、语速滑块设计就好
>
> **建议：站在巨人肩膀上造小轮子。** 不引入外部构建依赖，只从中提取设计思路。
>
> ```
> 前端：参考 AudioMass 的 Canvas 波形 + 选区拖拽（~80行 JS）
> TTS：pip install edge-tts（直接调用微软云，最省事）
> 后端：Python soundfile + numpy（已有）
> 保存：现有 bells 表 + static/sounds/（已有）
> 总代码量 ~300行，零构建负担
> ```

**王工（运维）——"稳定性和维护风险怎样？"**

> 逐一评估风险：
>
> **AudioMass（2.8k⭐）**：MIT 协议，代码质量高，194 commits，活跃维护。可以直接复制参考其中的 JS 代码（MIT 允许）。如果未来想升级，可以考虑 iframe 嵌入，但需要对它的 build 流程做适配。
>
> **waveform-playlist（1.6k⭐）**：**AGPL-3.0 协议**（有传染性！）。React 生态，npm 构建。两个硬伤：① AGPL 协议要求派生作品也必须 AGPL，不适合闭源部署 ② React 和 vanilla JS 不兼容。❌ 排除。
>
> **tts.rocks（30⭐）**：MIT 协议。纯前端，TTS 模型在浏览器加载。如果它在后台下载大模型，会消耗用户带宽。风险低但体验可能受影响。
>
> **其余项目（<10⭐）**：代码质量未经验证，不建议直接依赖，只参考实现思路。
>
> **风险评估表：**
>
> | 方案 | 风险 | 维护成本 | 适配度 | 推荐 |
> |------|------|---------|--------|------|
> | 直接集成 AudioMass | 中（iframe 通信复杂） | 高 | 60% | ❌ |
> | 直接集成 waveform-playlist | 高（AGPL+React） | 高 | 20% | ❌❌ |
> | **参考 + 自研轻量版** | **低** | **低（~300行）** | **100%** | **✅✅** |
>
> **结论和之前一致——自研轻量版本最稳妥。不引入外部依赖风险，代码量可控，完全掌控。**

### 结论

| 问题 | 回答 |
|------|------|
| **有现成可用的项目吗？** | ❌ 没有。没有任何项目满足"TTS+裁剪+混音+保存到服务器" |
| **有参考价值的项目吗？** | ✅ AudioMass（波形交互）、edge-tts（TTS引擎）、tts.rocks（TTS UI） |
| **推荐方案？** | **参考 + 自研轻量版**，前端参考 AudioMass Canvas，后端 edge-tts + soundfile + numpy |

| **新增依赖？** | 仅 `pip install edge-tts`，前端零新增依赖 |
| **总代码量？** | ~300 行（后端 ~150 + 前端 ~150） |
| **开发时间？** | 后端 ~2.5 小时，前端 ~3 小时 |
| **风险？** | 极低。不改变现有系统架构，不引入构建工具 |
