# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

## 五、预约铃声（独立页面，绑定 task 的临时换铃）

### 需求确认

- **有一个独立的页面叫"预约铃声"**，不是嵌入课表
- 页面加载课表数据，展示所有区域下的任务列表（作为选择来源）
- 用户在这个页面可以**创建多个预约**
- 每个预约：选一个课表任务 → 选日期范围 → 选替换铃声
- 打铃时：`task → 查 task_overrides 是否有今天生效的记录 → 有就用预约铃，没有用默认铃`

### 专家讨论

**张老师（产品）**：

> 独立页面是对的。课表页面是管理"规律性安排"的，预约页面是管理"临时变动"的，两者职责分开，用户不会混淆。
>
> 这个页面的布局应该这样：
>
> ```
> ┌─────────────────────────────────────────────┐
> │  📅 预约铃声管理                              │
> │  ┌─────────────────────────────────────────┐ │
> │  │  新建预约  [选择区域▼] [选择课表▼]       │ │
> │  ├─────────────────────────────────────────┤ │
> │  │  预约列表（按生效日期排序）               │ │
> │  │  ┌────┬──────┬────────┬──────┬──────┐  │ │
> │  │  │区域│ 任务 │ 替换铃 │ 有效期 │ 操作 │  │ │
> │  │  ├────┼──────┼────────┼──────┼──────┤  │ │
> │  │  │ A  │第2节 │考试提示│6.22  │ 删除 │  │ │
> │  │  │    │上课  │ 音     │-6.26 │      │  │ │
> │  │  │ B  │第4节 │消防演 │6.20  │ 删除 │  │ │
> │  │  │    │下课  │ 练     │(仅今天│      │  │ │
> │  │  └────┴──────┴────────┴──────┴──────┘  │ │
> │  │  (共 2 条预约)                           │ │
> └─────────────────────────────────────────────┘
> ```

**陈工（架构）**：

> 独立页面反而让后端更清晰。接口设计：
>
> **新增 3 条 API**：
>
> | 方法 | 路径 | 说明 |
> |------|------|------|
> | `GET` | `/api/overrides` | 查全部预约（支持 zone_id 过滤） |
> | `POST` | `/api/overrides` | 新建预约 |
> | `DELETE` | `/api/overrides/:id` | 删除预约（=恢复默认） |
>
> 新建预约请求体：
> ```json
> {
>   "task_id": 27,
>   "start_date": "2026-06-22",
>   "end_date": "2026-06-26",
>   "bell_id": 15
> }
> ```
>
> **查询预约列表响应**：
> ```json
> {
>   "overrides": [
>     {
>       "id": 1,
>       "task_id": 27,
>       "zone_id": "A",
>       "zone_name": "广播室",
>       "task_time": "08:00",
>       "task_name": "第2节上课",

>       "override_bell": "考试提示音",
>       "start_date": "2026-06-22",
>       "end_date": "2026-06-26"
>     }
>   ]

> ```
>
> 注意这里做了一次 JOIN，把 task 的详细信息（区域名、时间、任务名）和铃声名称都查出来，前端直接展示，不用再额外请求。
>
> **前端页面引入的数据**（页面加载时并行请求）：
> - `GET /api/zones` — 区域列表（用于筛选器）
> - `GET /api/schedules` — 课表列表（用于筛选器）
> - `GET /api/overrides` — 预约列表（主表格数据）
> - `GET /api/bells` — 铃声列表（新建弹窗的选择器）

**王工（运维）**：

## 六、数据持久化分析（断电重启后数据是否保持）

### 问题

用户问：**课表数据、预约铃声数据，断电重启后是否还在？**

### 专家讨论

**陈工（架构）**：

> 回答：**全部保持。** 现有系统全部数据都存放在 `data/school_bell.db`，这是一个 SQLite 数据库文件，存储在硬盘上。断电不丢失数据。
>
> 我来分三层说清楚：

#### 第一层：存储层（SQLite）

```
数据存放位置：
  C:\Users\server\Desktop\钢城智慧铃声系统\data\school_bell.db  ✓ 硬盘文件
```

现有数据全部在 SQLite 中：

| 数据 | 表名 | 是否持久 |
|------|------|---------|
| 区域配置（名称、音频设备、音量） | `zones` | ✅ 硬盘 |
| 课表方案 | `schedules` | ✅ 硬盘 |
| 课表任务（时间、铃型、星期） | `tasks` | ✅ 硬盘 |
| 铃声库（文件名、类型） | `bells` | ✅ 硬盘 |
| 今日例外、明日覆盖 | `today_exceptions`, `tomorrow_overrides` | ✅ 硬盘 |
| 节假日 | `holidays` | ✅ 硬盘 |
| 运行日志 | `logs` | ✅ 硬盘 |
| **新建：预约铃声** | `task_overrides` | ✅ **硬盘** |

SQLite 使用 WAL（Write-Ahead Logging）模式，具备**崩溃安全**特性。写入时先写 WAL 再写 DB，断电恢复时自动回放 WAL 或回滚未完成的事务。不会出现"写了一半断电导致数据损坏"的情况。

> **一句话：只要点过"保存"，数据就已经在硬盘上了。拔电源再开机，数据还在。**

#### 第二层：调度层（APScheduler）

APScheduler 是内存调度器，断电后**调度状态会丢失**，但它会在启动时重新从数据库读取课表任务重建调度。

重启时序：

```
① 系统开机
② run.py 启动
③ init_system() 执行
   ├── init_db() → 打开 school_bell.db ✓
   ├── _load_zone_devices() → 读 zones 表 → zone_devices ✓
   └── scheduler.start() → APScheduler 启动
④ bell_check_task 开始每秒执行一次
   └── 读 tasks 表 → 匹配当前时间 → 打铃
```

**重启后确实会丢掉一些信息：**

| 会丢的 | 原因 | 影响 |
|--------|------|------|
| 正在播放中的铃声 | 铃声在内存中播放，断电即停 | 重新开机后不会继续播 |
| 已执行过的打铃记录 | `logs` 表持久，但本次 session 的日志缓冲区可能没刷盘 | 差最后几秒的日志 |
| 定时器"剩余等待时间" | APScheduler 下次触发时间是内存中的 | 重启后重新计算 |

**不会丢的：**

| 不会丢的 | 原因 |
|----------|------|
| 课表任务（含新增、修改） | 全在 SQLite |
| 区域配置 | 全在 SQLite |
| 铃声文件（WAV/MP3） | 全在硬盘 `static/sounds/` |
| **预约铃声记录** | 全在 SQLite `task_overrides` 表 |
| 今日例外、明日覆盖 | 全在 SQLite |

**边界情况——停电期间错过的铃声怎么办？**

假设停电发生在 08:00 - 08:30：

| 时间 | 事件 | 开机后 |
|------|------|--------|
| 07:55 | 正常打铃 ✅ | — |
| 08:00 | ⚡断电，第2节上课铃没响 | 08:31 恢复供电，系统启动 |
| 08:05-08:30 | 停电中，所有铃都没响 | 不补打（已错过） |
| 08:31 | 系统启动 | 判断当前时间 08:31，无匹配任务 → 等待下一节课 |
| 08:50 | 第3节上课铃 | ✅ 正常响起 |

> 结论：**错过的铃不补打**，之后恢复正常。学校场景下这是可接受的——电都停了，打了也没人听。

#### 第三层：预约铃声持久化（新建功能）

`task_overrides` 表和现有表完全一样存放在 SQLite 中：

```sql
CREATE TABLE task_overrides (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    INTEGER NOT NULL,
    start_date TEXT NOT NULL,
    end_date   TEXT NOT NULL,
    bell_id    INTEGER NOT NULL,
    desc       TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
```

新建预约时：
```
POST /api/overrides → INSERT INTO task_overrides → conn.commit() → 落盘 ✓
```

重启后：
```
bell_check_task → SELECT * FROM task_overrides WHERE task_id=? AND ? BETWEEN start_date AND end_date
                → 能查到之前建的记录 ✓ → 正常使用预约铃
```

**预约铃声和普通课表任务在持久化上没有任何区别。**

### 关键定论

| 问题 | 回答 |
|------|------|
| 课表数据断电会丢吗？ | ❌ 不会。全部在 SQLite 硬盘文件 |
| 预约铃声数据断电会丢吗？ | ❌ 不会。和课表同一张 DB |
| 正在播放的铃断电会续吗？ | ❌ 不会续播（断电时中断即丢，开机后不补） |
| 停电期间错过的铃会补吗？ | ❌ 不补。恢复后从当前时间继续 |
| 数据会损坏吗？ | ✅ SQLite WAL 模式保证崩溃安全 |
| 是否需要 UPS？ | 可选。不是必须，但有更好（能覆盖短暂停电） |
