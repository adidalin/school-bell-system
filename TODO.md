# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

### 状态

> ⬜ 未开始

## 五、预约铃声（独立页面，绑定 task 的临时换铃）

### 需求确认

- **有一个独立的页面叫"预约铃声"**，不是嵌入课表
- 页面加载课表数据，展示所有区域下的任务列表（作为选择来源）
- 用户在这个页面可以**创建多个预约**
- 每个预约：选一个课表任务 → 选日期范围 → 选替换铃声
- 打铃时：`task → 查 task_overrides 是否有今天生效的记录 → 有就用预约铃，没有用默认铃`

### 专家讨论

**张老师（产品）**：

> 独立页面是对的。课表页面是管理"规律性安排"的，预约页面是管理"临时变动"的，两者职责分开，用户不会混淆。
>
> 这个页面的布局应该这样：
>
> ```
> ┌─────────────────────────────────────────────┐
> │  📅 预约铃声管理                              │
> │  ┌─────────────────────────────────────────┐ │
> │  │  新建预约  [选择区域▼] [选择课表▼]       │ │
> │  ├─────────────────────────────────────────┤ │
> │  │  预约列表（按生效日期排序）               │ │
> │  │  ┌────┬──────┬────────┬──────┬──────┐  │ │
> │  │  │区域│ 任务 │ 替换铃 │ 有效期 │ 操作 │  │ │
> │  │  ├────┼──────┼────────┼──────┼──────┤  │ │
> │  │  │ A  │第2节 │考试提示│6.22  │ 删除 │  │ │
> │  │  │    │上课  │ 音     │-6.26 │      │  │ │
> │  │  │ B  │第4节 │消防演 │6.20  │ 删除 │  │ │
> │  │  │    │下课  │ 练     │(仅今天│      │  │ │
> │  │  └────┴──────┴────────┴──────┴──────┘  │ │
> │  │  (共 2 条预约)                           │ │
> └─────────────────────────────────────────────┘
> ```

**陈工（架构）**：

> 独立页面反而让后端更清晰。接口设计：
>
> **新增 3 条 API**：
>
> | 方法 | 路径 | 说明 |
> |------|------|------|
> | `GET` | `/api/overrides` | 查全部预约（支持 zone_id 过滤） |
> | `POST` | `/api/overrides` | 新建预约 |
> | `DELETE` | `/api/overrides/:id` | 删除预约（=恢复默认） |
>
> 新建预约请求体：
> ```json
> {
>   "task_id": 27,
>   "start_date": "2026-06-22",
>   "end_date": "2026-06-26",
>   "bell_id": 15
> }
> ```
>
> **查询预约列表响应**：
> ```json
> {
>   "overrides": [
>     {
>       "id": 1,
>       "task_id": 27,
>       "zone_id": "A",
>       "zone_name": "广播室",
>       "task_time": "08:00",
>       "task_name": "第2节上课",
>       "original_bell": "上课铃声",
>       "override_bell": "考试提示音",
>       "start_date": "2026-06-22",
>       "end_date": "2026-06-26"
>     }
>   ]
> }
> ```
>
> 注意这里做了一次 JOIN，把 task 的详细信息（区域名、时间、任务名）和铃声名称都查出来，前端直接展示，不用再额外请求。
>
> **前端页面引入的数据**（页面加载时并行请求）：
> - `GET /api/zones` — 区域列表（用于筛选器）
> - `GET /api/schedules` — 课表列表（用于筛选器）
> - `GET /api/overrides` — 预约列表（主表格数据）
> - `GET /api/bells` — 铃声列表（新建弹窗的选择器）

**王工（运维）**：

> 多预约并存的情况下，会不会出现"同一个 task 在同一个日期有两条预约"？
>
> 不应该出现。数据库层面加唯一约束：
> ```sql
> CREATE UNIQUE INDEX idx_task_date ON task_overrides(task_id, start_date);
> ```
>
> 或者如果允许同 task 同日期但不同时段... 这个模型里不需要，因为一个 task 就是一个固定时间点。同一天同一 task 不可能有两个不同的预约，逻辑上冲突。
>
> 前端也要做校验：用户选了 task + 日期范围后，检查是否已有冲突预约，如果有就提示"该任务在选定的日期范围内已有预约"。

**小花荣（用户）**：

> 我确认一下我理解的对不对：
>
> 1. 我打开"预约铃声"页面
> 2. 看到一大张表，列出了所有区域所有课表的所有任务
> 3. 我找到"小学→周一课表→第2节上课"这一行，点"预约"
> 4. 弹窗：日期范围选6.22-6.26，铃声选"考试提示音"
> 5. 保存 → 预约列表里多了一条记录
> 6. 6月22日到26日之间，每天第2节上课时播放"考试提示音"
> 7. 6月27日起自动恢复默认
>
> **陈工**：完全正确。

**张老师总结**：

> 这个设计可行。和之前的结论一样——`bell_check_task` 改 5 行，其他全是新增。独立页面比嵌入课表更好维护，两个页面职责分开，用户不迷惑。

### 关键定论

| 问题 | 回答 |
|------|------|
| 页面 | 独立"预约铃声"页面，不是嵌入课表 |
| 数据绑定 | 每个预约绑定一个 `task_id`，不同预约互不冲突 |
| 多预约 | ✅ 可以建任意多个，各自独立生效 |
| 冲突检测 | DB 唯一索引 + 前端校验，同 task+同日期不允许多条 |
| 打铃改动 | `bell_check_task` 改 **5 行**：查 `task_overrides` → 命中就换铃 |
| 新增 API | 3 条：查列表 / 新建 / 删除 |
| 新增表 | `task_overrides` 一张 |
| 总代码量 | ~180 行（后端 ~60，前端 ~120） |
| 开发时间 | 后端 ~1.5 小时，前端 ~2 小时 |
