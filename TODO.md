# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

### 状态

> ⬜ 未开始

## 五、预约铃声（绑定到 task 的临时换铃）

### 用户模型（确认版）

> 不建新页面。在课表页面里，每个任务行右边加一个"预约"按钮。
>
> 点按钮 → 弹窗选：① 响铃日期范围（哪几天生效） ② 替换哪个铃声
>
> 保存后，在这个日期范围内到了这个时间点，就播放预约的铃声；没有预约就用课表原来的铃声。
>
> **每次响铃之前加一个判断。**

### 专家讨论

**张老师（产品）**：

> 你这个模型比"区域+类型+时间"匹配直观太多了。老师看课表就看到任务，"第2节上课"这个任务旁边点预约，选哪几天换什么铃——不用理解什么 bell_type、匹配规则。这是老师能直接用的设计。
>
> 使用场景还是那些：
> - 下周一到周五，第2节上课换成考试提示音
> - 仅今天，第3节下课换成消防演练音
> - 下个月整个月，第1节上课换成校歌
>
> 但在你的模型下，这些操作就是：找到对应任务行 → 点预约 → 选日期和铃声。三步完成。

**陈工（架构）**：

> 你这个模型比我的原方案**简单一倍**。我仔细对比：

| 对比项 | 我之前的设计（zone+type+time） | 你的设计（绑定 task_id） |
|--------|-------------------------------|------------------------|
| 数据量 | 按区域、铃型、时间三重匹配，可能命中多条需优先级裁决 | 查 `task_id` 直接命中，最多一条 |
| 匹配逻辑 | `WHERE zone_id=? AND bell_type=? AND time>=? AND ...` 复杂的组合查询 | `WHERE task_id=? AND start_date<=today AND end_date>=today` 简单精确 |
| 前端入口 | 需要新建一个独立页面，用户要填很多字段 | **课表任务行旁点按钮即可**，弹窗只选日期+铃声 |
| 对现有系统入侵 | 改 bell_check_task 的匹配逻辑 | **同样只改 5 行** |
| 误操作风险 | 用户可能填错区域/时间匹配到不该换的铃声 | 绑定到具体 task，不可能越界 |

> **数据模型**也简化了：
>
> ```sql
> CREATE TABLE task_overrides (
>     id         INTEGER PRIMARY KEY AUTOINCREMENT,
>     task_id    INTEGER NOT NULL,       -- 绑定到具体课表任务
>     start_date TEXT NOT NULL,          -- 生效起始日期
>     end_date   TEXT NOT NULL,          -- 生效截止日期
>     bell_id    INTEGER NOT NULL,       -- 替换为哪个铃声
>     desc       TEXT,                   -- 备注
>     created_at TEXT DEFAULT (datetime('now'))
> );
> ```
>
> **打铃判断逻辑**（改 `bell_check_task` 的 5 行）：
>
> ```python
> # 当前流程（伪代码）
> for task in matching_tasks:
>     bell = get_bell_by_type(task.bell_type)  # 查默认铃声
>     player.play(zone, bell.filepath)
>
> # 改后流程
> for task in matching_tasks:
>     override = db.execute(
>         "SELECT bell_id FROM task_overrides WHERE task_id=? AND ? BETWEEN start_date AND end_date",
>         [task.id, today]
>     ).fetchone()
>     bell_id = override.bell_id if override else task.bell_type  # 有预约用预约，否则用默认
>     bell = get_bell_by_id_or_type(bell_id)
>     player.play(zone, bell.filepath)
> ```
>
> 注意 `task.bell_type` 存的是字符串（如 "class_start"），而 `override.bell_id` 存的是整数 ID。所以 `get_bell_by_id_or_type` 需要同时支持两种查询方式——用类型名或 ID 找铃声文件。这是唯一需要兼容的地方。

**王工（运维）**：

> task_id 绑定模型还有个好处：**删除 task 时级联清理 override**。
>
> 如果老师删除了一个课表任务，对应的预约记录就成了"孤儿数据"。可以用 SQLite 的外键 `ON DELETE CASCADE` 自动清理：
>
> ```sql
> CREATE TABLE task_overrides (
>     ...
>     task_id INTEGER NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
>     ...
> );
> ```
>
> 这样课表删任务时，预约记录自动跟着删，不用额外写清理逻辑。
>
> 另外，预约的日期范围应该有个上限——不允许跨学年（比如不能从2026年6月预约到2028年），前端可以限制最多选 60 天。

**小花荣（用户）**：

> 我想确认一下操作流程：
>
> 1. 我打开课表页面，看到"第2节上课"这一行
> 2. 点旁边的"预约🔔"按钮
> 3. 弹窗里我选：从6月22日到6月26日（周一到周五），铃声选"考试提示音"
> 4. 保存
> 5. 这周内每天第2节上课时，自动播放"考试提示音"
> 6. 6月27日起，自动恢复为课表原来的铃声
>
> 是不是这么简单？
>
> **陈工**：完全正确。到期自动恢复，不需要手动干预。

### 关键定论

| 问题 | 回答 |
|------|------|
| 改动大吗？ | **极小。** `bell_check_task` 改 **5 行**，其余全是新增代码 |
| 数据模型 | 新增 `task_overrides` 表，外键级联删除 |
| 影响现有系统？ | ❌ 无影响。无预约时走原路径，性能零损耗 |
| SQL 复杂度 | `WHERE task_id=? AND ? BETWEEN start_date AND end_date`，唯一索引保障 |
| 新增 API | `POST/GET/DELETE /api/tasks/:id/override` 共 3 条 |
| 前端改动 | 课表任务行加"预约"按钮 + 弹窗（日期选择器 + 铃声下拉），约 100 行 |
| 后端新增 | 表迁移 ~10 行 + API 路由 ~40 行 + 调度 5 行 |
| 总代码量 | ~155 行（后端 ~55，前端 ~100） |
| 开发时间 | 后端 ~1 小时，前端 ~2 小时 |
