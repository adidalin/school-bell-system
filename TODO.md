## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

## 六、数据持久化分析（断电重启后数据是否保持）

### 问题

用户问：**课表数据、预约铃声数据，断电重启后是否还在？**

### 专家讨论

**陈工（架构）**：

> 回答：**全部保持。** 现有系统全部数据都存放在 `data/school_bell.db`，这是一个 SQLite 数据库文件，存储在硬盘上。断电不丢失数据。
>
> 我来分三层说清楚：

#### 第一层：存储层（SQLite）

```
数据存放位置：
  C:\Users\server\Desktop\钢城智慧铃声系统\data\school_bell.db  ✓ 硬盘文件
```

现有数据全部在 SQLite 中：

| 数据 | 表名 | 是否持久 |
|------|------|---------|
| 区域配置（名称、音频设备、音量） | `zones` | ✅ 硬盘 |
| 课表方案 | `schedules` | ✅ 硬盘 |
| 课表任务（时间、铃型、星期） | `tasks` | ✅ 硬盘 |
| 铃声库（文件名、类型） | `bells` | ✅ 硬盘 |
| 今日例外、明日覆盖 | `today_exceptions`, `tomorrow_overrides` | ✅ 硬盘 |
| 节假日 | `holidays` | ✅ 硬盘 |
| 运行日志 | `logs` | ✅ 硬盘 |
| **新建：预约铃声** | `task_overrides` | ✅ **硬盘** |

SQLite 使用 WAL（Write-Ahead Logging）模式，具备**崩溃安全**特性。写入时先写 WAL 再写 DB，断电恢复时自动回放 WAL 或回滚未完成的事务。不会出现"写了一半断电导致数据损坏"的情况。

> **一句话：只要点过"保存"，数据就已经在硬盘上了。拔电源再开机，数据还在。**

#### 第二层：调度层（APScheduler）

APScheduler 是内存调度器，断电后**调度状态会丢失**，但它会在启动时重新从数据库读取课表任务重建调度。

重启时序：

```
① 系统开机
② run.py 启动
③ init_system() 执行
   ├── init_db() → 打开 school_bell.db ✓
   ├── _load_zone_devices() → 读 zones 表 → zone_devices ✓
   └── scheduler.start() → APScheduler 启动
④ bell_check_task 开始每秒执行一次
   └── 读 tasks 表 → 匹配当前时间 → 打铃
```

**重启后确实会丢掉一些信息：**

| 会丢的 | 原因 | 影响 |
|--------|------|------|
| 正在播放中的铃声 | 铃声在内存中播放，断电即停 | 重新开机后不会继续播 |
| 已执行过的打铃记录 | `logs` 表持久，但本次 session 的日志缓冲区可能没刷盘 | 差最后几秒的日志 |
| 定时器"剩余等待时间" | APScheduler 下次触发时间是内存中的 | 重启后重新计算 |

**不会丢的：**

| 不会丢的 | 原因 |
|----------|------|
| 课表任务（含新增、修改） | 全在 SQLite |
| 区域配置 | 全在 SQLite |
| 铃声文件（WAV/MP3） | 全在硬盘 `static/sounds/` |
| **预约铃声记录** | 全在 SQLite `task_overrides` 表 |
| 今日例外、明日覆盖 | 全在 SQLite |

**边界情况——停电期间错过的铃声怎么办？**

假设停电发生在 08:00 - 08:30：

| 时间 | 事件 | 开机后 |
|------|------|--------|
| 07:55 | 正常打铃 ✅ | — |
| 08:00 | ⚡断电，第2节上课铃没响 | 08:31 恢复供电，系统启动 |
| 08:05-08:30 | 停电中，所有铃都没响 | 不补打（已错过） |
| 08:31 | 系统启动 | 判断当前时间 08:31，无匹配任务 → 等待下一节课 |
| 08:50 | 第3节上课铃 | ✅ 正常响起 |

> 结论：**错过的铃不补打**，之后恢复正常。学校场景下这是可接受的——电都停了，打了也没人听。

#### 第三层：预约铃声持久化（新建功能）

`task_overrides` 表和现有表完全一样存放在 SQLite 中：

```sql
CREATE TABLE task_overrides (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    INTEGER NOT NULL,
    start_date TEXT NOT NULL,
    end_date   TEXT NOT NULL,
    bell_id    INTEGER NOT NULL,
    desc       TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
```

新建预约时：
```
POST /api/overrides → INSERT INTO task_overrides → conn.commit() → 落盘 ✓
```

重启后：
```
bell_check_task → SELECT * FROM task_overrides WHERE task_id=? AND ? BETWEEN start_date AND end_date
                → 能查到之前建的记录 ✓ → 正常使用预约铃
```

**预约铃声和普通课表任务在持久化上没有任何区别。**

## 七、铃声编辑功能——简化方案讨论（裁剪去掉，只保留 TTS 前置语音）

### 用户提出的新模型

> 裁剪太复杂就不做了。只保留 TTS。用户想要的效果是：**打铃时先播放一段 TTS 语音，然后播放原来的铃声。** 不需要合并成一个文件，播放器顺序播放即可。

### 两种可选实现方案

| 方案 | 做法 | 工作量 | 复杂度 |
|------|------|--------|--------|
| **A. 运行时顺序播放** | 到点时：`play(TTS音频) → wait() → play(原铃声)` | ~30 行 | ⭐ 最简单 |
| **B. 预生成合并文件** | 配置时：`TTS音频 + 原铃声 = 拼接为一个新 WAV 文件` | ~60 行 | ⭐⭐ 稍复杂 |

**专家分组讨论：**

**陈工（架构）——"技术上好不好弄？"**

> 两个方案都非常好弄。

**方案 A（运行时顺序播放）——改 engine.py 的 play_zone 方法：**

```python
# 当前逻辑（简化）
def play_zone(zone, bell):
    player.play(bell.filepath)

# 改后逻辑
def play_zone(zone, bell, tts_text=None):
    if tts_text:
        tts_path = generate_tts(tts_text)  # 生成 TTS WAV 文件
        player.play(tts_path)              # 先播 TTS
        player.wait_until_done()           # 等 TTS 播完
    player.play(bell.filepath)             # 再播原铃声
```

这个方案的好处：
- **不需要数据库存新铃声**。TTS 文本存在 `task_overrides` 表（加一列 `tts_preamble TEXT`）或 `tasks` 表（加一列）。
- **不需要创建新文件**。TTS 临时生成、临时播放、播完删除。
- **零前端 UI**。预约铃声页面加一个"TTS 前置语音"文本框即可。
- **TTS 失败不影响原铃声**。`try/except` 兜底——TTS 生成失败就跳过，直接播原铃声。

风险点：
> TTS 生成需要网络（edge-tts）。但只在配置时生成一次、缓存起来，打铃时不再需要网络。或者按我下面的优化方案：**配置时预生成 TTS 文件，打铃时只负责播放**。

**方案 B（预生成拼接文件）——配置时一次性合成：**

```python
# 用户点"保存"时执行一次
tts_data, sr = load_tts_as_numpy(tts_text)    # edge-tts → numpy
bell_data, sr = sf.read(bell_path)             # 原铃声
combined = np.concatenate([tts_data, bell_data])  # 拼接
sf.write(final_path, combined, sr)             # 写出新文件
# → INSERT INTO bells (name, filepath, ...)
```

好处：打铃时 0 延迟，TTS+铃声音质完美。
坏处：每次修改 TTS 文本都要重新生成一个新铃声文件，`bells` 表会多出很多记录。

**我的推荐：方案 A 的优化版——缓存模式**

```
用户配置 TTS 文本
       ↓
保存时：生成 TTS WAV → 缓存到 temp/tts_cache/{id}.wav
       ↓
打铃时：播放缓存 TTS → 播完 → 播原铃声
       ↓
用户修改 TTS 文本 → 重新生成缓存，覆盖旧文件
```

这样打铃时不需要网络（文件已缓存），也不需要创建新铃声记录。TTS 文本存在 `task_overrides.tts_preamble` 字段即可。

| 步骤 | 代码量 |
|------|--------|
| DB migration（加一列 `tts_preamble`） | 5 行 |
| API 增改（读取/写入 TTS 文本） | 15 行 |
| TTS 生成 + 缓存逻辑 | 25 行 |
| engine.py 播放加前置 TTS | 10 行 |
| 前端加文本框 | 15 行 |
| **总计** | **~70 行** |

> 注意：这个模式和预约铃声是**天然配合**的。预约铃声是"换铃"，TTS 前置语音是"加前奏"。两个可以叠加：预约换铃 + TTS 前置 → 先播 TTS，再播预约的替换铃声。

**王工（运维）——"会不会引入新 bug？"**

> 我逐个检查风险点：
>
> **TTS 生成失败（网络问题）：**
> ```python
> try:
>     tts_path = generate_tts(text)
> except Exception:
>     tts_path = None  # 静默跳过，不影响原铃声
> 
> if tts_path:
>     player.play(tts_path)
>     player.wait_until_done()
> player.play(bell_path)
> ```
> 
> TTS 生成失败唯一的结果是"只播原铃声，没有前置语音"。**不会导致不打铃、不会抛异常中断打铃线程、不会影响其他区域。**
>
> **TT 缓存文件未清理：**
> 缓存文件放在 `temp/tts_cache/`，文件名用 task_id 做 hash。每次重新生成覆盖旧文件。长期不用就留在那里，几个 KB 不占空间。如果非要清理，可以定时扫描 `task_overrides` 表，删除没有引用的缓存。
>
> **异常风险矩阵：**
>
> | 风险 | 概率 | 影响 | 应对 |
> |------|------|------|------|
> | TTS 生成时断网 | 低 | 跳过 TTS，只播原铃声 | try/except 兜底 |
> | TTS 生成超时（>15s） | 极低 | 同上 | 设置 timeout |
> | TTS 缓存文件损坏 | 极低 | 同上 | 播放前检测文件有效性 |
> | TTS 文本过长 | 低 | 语音太长，耽误打铃 | 限制 100 字以内 |
> | 和预约铃声同时使用时冲突 | 无 | 两者叠加，先 TTS 再播预约铃 | 逻辑上天然兼容 |
>
> **结论：零新增风险。** 所有异常都有兜底——TTS 失败就跳过，铃声照常响。比全功能音频编辑器的风险低一个数量级。
