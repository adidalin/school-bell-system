## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

## 六、数据持久化分析（断电重启后数据是否保持）

### 问题

用户问：**课表数据、预约铃声数据，断电重启后是否还在？**

### 专家讨论

**陈工（架构）**：

> 回答：**全部保持。** 现有系统全部数据都存放在 `data/school_bell.db`，这是一个 SQLite 数据库文件，存储在硬盘上。断电不丢失数据。
>
> 我来分三层说清楚：

#### 第一层：存储层（SQLite）

```
数据存放位置：
  C:\Users\server\Desktop\钢城智慧铃声系统\data\school_bell.db  ✓ 硬盘文件
```

现有数据全部在 SQLite 中：

| 数据 | 表名 | 是否持久 |
|------|------|---------|
| 区域配置（名称、音频设备、音量） | `zones` | ✅ 硬盘 |
| 课表方案 | `schedules` | ✅ 硬盘 |
| 课表任务（时间、铃型、星期） | `tasks` | ✅ 硬盘 |
| 铃声库（文件名、类型） | `bells` | ✅ 硬盘 |
| 今日例外、明日覆盖 | `today_exceptions`, `tomorrow_overrides` | ✅ 硬盘 |
| 节假日 | `holidays` | ✅ 硬盘 |
| 运行日志 | `logs` | ✅ 硬盘 |
| **新建：预约铃声** | `task_overrides` | ✅ **硬盘** |

SQLite 使用 WAL（Write-Ahead Logging）模式，具备**崩溃安全**特性。写入时先写 WAL 再写 DB，断电恢复时自动回放 WAL 或回滚未完成的事务。不会出现"写了一半断电导致数据损坏"的情况。

> **一句话：只要点过"保存"，数据就已经在硬盘上了。拔电源再开机，数据还在。**

#### 第二层：调度层（APScheduler）

APScheduler 是内存调度器，断电后**调度状态会丢失**，但它会在启动时重新从数据库读取课表任务重建调度。

重启时序：

```
① 系统开机
② run.py 启动
③ init_system() 执行
   ├── init_db() → 打开 school_bell.db ✓
   ├── _load_zone_devices() → 读 zones 表 → zone_devices ✓
   └── scheduler.start() → APScheduler 启动
④ bell_check_task 开始每秒执行一次
   └── 读 tasks 表 → 匹配当前时间 → 打铃
```

**重启后确实会丢掉一些信息：**

| 会丢的 | 原因 | 影响 |
|--------|------|------|
| 正在播放中的铃声 | 铃声在内存中播放，断电即停 | 重新开机后不会继续播 |
| 已执行过的打铃记录 | `logs` 表持久，但本次 session 的日志缓冲区可能没刷盘 | 差最后几秒的日志 |
| 定时器"剩余等待时间" | APScheduler 下次触发时间是内存中的 | 重启后重新计算 |

**不会丢的：**

| 不会丢的 | 原因 |
|----------|------|
| 课表任务（含新增、修改） | 全在 SQLite |
| 区域配置 | 全在 SQLite |
| 铃声文件（WAV/MP3） | 全在硬盘 `static/sounds/` |
| **预约铃声记录** | 全在 SQLite `task_overrides` 表 |
| 今日例外、明日覆盖 | 全在 SQLite |

**边界情况——停电期间错过的铃声怎么办？**

假设停电发生在 08:00 - 08:30：

| 时间 | 事件 | 开机后 |
|------|------|--------|
| 07:55 | 正常打铃 ✅ | — |
| 08:00 | ⚡断电，第2节上课铃没响 | 08:31 恢复供电，系统启动 |
| 08:05-08:30 | 停电中，所有铃都没响 | 不补打（已错过） |
| 08:31 | 系统启动 | 判断当前时间 08:31，无匹配任务 → 等待下一节课 |
| 08:50 | 第3节上课铃 | ✅ 正常响起 |

> 结论：**错过的铃不补打**，之后恢复正常。学校场景下这是可接受的——电都停了，打了也没人听。

#### 第三层：预约铃声持久化（新建功能）

`task_overrides` 表和现有表完全一样存放在 SQLite 中：

```sql
CREATE TABLE task_overrides (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    INTEGER NOT NULL,
    start_date TEXT NOT NULL,
    end_date   TEXT NOT NULL,
    bell_id    INTEGER NOT NULL,
    desc       TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
```

新建预约时：
```
POST /api/overrides → INSERT INTO task_overrides → conn.commit() → 落盘 ✓
```

重启后：
```
bell_check_task → SELECT * FROM task_overrides WHERE task_id=? AND ? BETWEEN start_date AND end_date
                → 能查到之前建的记录 ✓ → 正常使用预约铃
```

**预约铃声和普通课表任务在持久化上没有任何区别。**

**王工（运维）**：

> 我补充一个运维层面的建议：

> 2. **启动日志**：每次启动在日志里记一条 `System started after power loss`，方便排查

> 用户问的核心问题就是"我辛辛苦苦配的课表、建的预约，断电还有没有"——答案是 **有，全在硬盘里。** 拔电重启跟关掉浏览器再打开一样，数据丝毫无损。放心用。

### 关键定论

| 问题 | 回答 |
|------|------|
| 课表数据断电会丢吗？ | ❌ 不会。全部在 SQLite 硬盘文件 |

|------|---------|---------|
| 导入已有铃声 | `sf.read(path)` → numpy array | 1 |
| 裁剪（截取片段） | `data[start:end]` | 1 |
| TTS 生成语音段 | `pyttsx3.save_to_file()` → `sf.read()` | 3 |
| 混合（声音叠加） | `np.add(data1, data2) / clip()` | 3 |
| 拼接（先后播放） | `np.concatenate([data1, data2])` | 1 |
| 音量调整 | `data * factor` | 1 |
| 淡入淡出 | `data * fade_curve` | 3 |
| 导出新铃声文件 | `sf.write(path, data, sr)` → 写入 `static/sounds/` | 1 |

**关键细节：**

> 1. **WAV 是唯一格式**。soundfile 在 Windows 上自带 libsndfile，支持 WAV/FLAC/OGG。TTS 输出 WAV，裁剪保留 WAV，最终存 WAV。系统已有的铃声也是 WAV，完美兼容。
>
> 2. **采样率统一**。系统现有铃声采样率 22050，TTS 可能输出 16000 或 22050。混音前必须重采样到一致。Python 实现：`scipy.signal.resample` 可以，但不一定装了；更简单的方法：用 `np.interp` 或直接 `sf.read(..., samplerate=22050)` 让 soundfile 自动处理。
>
> 3. **中文 TTS**。Windows 10/11 内置"Microsoft Huihui"简体中文语音。pyttsx3 可以通过 `engine.setProperty('voice', voice.id)` 选中。效果清晰自然，适合铃声提示语（如"考试时间到，请考生停止答题"）。
>
> 4. **编辑后生成的是新的铃声文件**。不修改原文件。流程：
> ```
> 用户上传铃声.wav ↘
>                   → numpy 处理 → static/sounds/edited_xxx.wav
> 用户输入文字   → TTS → WAV ↗                ↓
>                                         bells 表新增一条记录 ← 用户命名
> ```

**前端实现思路：**

由于本项目没有前端构建工具（纯 HTML + JS + CSS），波形显示有两种方案：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A. 后端渲染波形图** | Flask 用 matplotlib 生成波形 PNG，前端 img 显示 | 实现最快，不依赖前端库 | 不能交互拖拽裁剪 |
| **B. 前端 Canvas 绘制** | API 返回波形峰值数据（~1000个float），前端 Canvas draw | 可以交互拖拽裁剪 | 稍多一些 JS 代码 |

推荐**方案 B**：后端返回 `GET /api/bell-editor/waveform?file=xxx` → `{peaks: [...], duration: 12.5}`，前端用 Canvas 画波形 + 拖拽滑块设置起止点。代码量约 80 行 JS。

**张老师（产品）**：

> 这个功能的用户场景是什么？
>
> 场景 1：教导主任想播一段"考试时间到，请考生停止答题"的语音提示，但现有的铃声库没有。她不需要找外部工具合成，直接在系统里输入文字 → TTS 生成 → 保存为铃声 → 设置为某个任务的铃声。**全在一个系统里完成**。
>
> 场景 2：现有的上课铃声太短，她想在前面加一段"同学们好"，但不想重新找音频文件。导入原铃声 → 裁剪保留后段 → TTS 生成"同学们好" → 拼接在一起 → 保存为新铃声。
>
> 场景 3：她有一段现成的 WAV 音频，只想截取中间 10 秒作为铃声。导入 → 拖拽选区 → 导出为新铃声。
>
> 总结：**这个功能解决的是"不想离开本系统去外部工具编辑音频"的问题**。对学校用户来说，少一个工具切换就少一层学习成本。
