# 后期开发计划 & 待解决问题

## 一、PocketBase 权限系统集成

### 目标
Flask 接入 PocketBase 做鉴权，实现角色级数据隔离。

### 角色与权限

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| `admin` | 全部区域 | 全部权限 |
| `primary`（小学） | 仅小学区域 | 编辑课表、上传铃声、查看日志 |
| `middle`（初中） | 仅初中区域 | 编辑课表、上传铃声、查看日志 |
| `uploader` | 无区域 | 仅上传铃声 |

### 实现要点

- PocketBase 与 Flask 同机运行，端口 `8090`
- Flask 加 JWT 中间件，验证每个请求的令牌
- 用户表扩展 `zone_id` 和 `role` 字段
- 前端加登录页，路由守卫
- `run.py` 启动时自动拉起 PocketBase 子进程

### 不改的部分

- 不打铃调度（调度逻辑不变）
- 不迁移数据（PocketBase 仅存用户，区域/课表/铃声仍在 SQLite）
- 不改造音频引擎

### 状态

> ⬜ 未开始

## 五、预约铃声（铃音临时覆盖）

### 专家讨论

**张老师（产品）**：

> 这个需求很实在。课表是固定的框架，但实际教学中有太多临时变动——下周要考试了，上课铃想换成"考生入场"提示音；或者这一周搞活动，下课铃改成轻音乐。老师不可能为了这个去改整个课表。
>
> 核心逻辑就是一句话：**打铃前先查有没有预约，有就用预约的，没有就用课表默认的。**
>
> 使用场景：
> - 下周一到周五，把上午第二节下课铃换成指定铃声
> - 仅今天，把全校的上课铃换成消防演练提示音
> - 下个月整个月，把初中部的放学铃换成校歌

**陈工（架构）**：

> 先回答用户最关心的问题：**改动大不大？答案是不大。**
>
> 我拆开来说为什么不大：
>
> **第一，只改一个函数里的 5 行。**
>
> 打铃的主逻辑在 `engine.py` 的 `bell_check_task`（APScheduler 每秒执行一次）。当前流程：
>
> ```
> ② 找到匹配的 task → ③ task.bell_type → ④ play(bell_type对应的WAV)
> ```
>
> 改成：
>
> ```
> ② 找到匹配的 task → ③ task.bell_type → ③′ 查 overrides 表
>                                        → 有命中? 用 override.bell_id
>                                        → 无命中? 用 task 默认 bell_type
>                                        → ④ play(最终确定的WAV)
> ```
>
> ③′ 这一步是新增的 5 行代码：一条 SQL 查询 + 一个 if 判断。不涉及音频引擎、不涉及多线程、不涉及混音逻辑。
>
> **第二，其余代码都是"新增"不是"改动"。**
>
> | 代码 | 性质 | 行数 |
> |------|------|------|
> | DB 迁移（CREATE TABLE） | 新文件末尾追加 | ~15 |
> | `models.py` 查预约函数 | 新增函数 | ~10 |
> | API 路由 4 条 | 新增函数 | ~60 |
> | 前端预约页面 | 新增 HTML 片段 | ~150 |
> | bell_check_task 加 5 行 | **唯一改动** | **~5** |
>
> **第三，不碰任何现有逻辑。**
>
> - 不改 task 表结构
> - 不改 schedule 表结构
> - 不改 zones 表结构
> - 不改 bells 表结构
> - 不改 audio player
> - 不改 设备枚举
> - 不改 前端课表页面
>
> 预约表是一个完全独立的附属表，不影响任何现有查询和写入。没有预约记录时，`bell_check_task` 走的就是原来的路径，性能零损耗。

**王工（运维）**：

> 我补充一个担心：overrides 表如果积累了上万条过期记录，每秒查一次会不会慢？
>
> SQLite 在这种场景下没有问题。`SELECT * FROM bell_overrides WHERE zone_id=? AND bell_type=? AND date>=start_date AND date<=end_date` 在几万行数据下是微秒级的。更不用说加个索引后基本没感觉。
>
> 自动清理还是要有——每天凌晨跑一次 `DELETE FROM bell_overrides WHERE end_date < date('now')`，保持表干净。

**小花荣（用户）**：

> 那如果我先预约了下周用铃声A，过了两天我又想改回默认了，是不是要把预约删掉？
>
> **陈工**：对。删除预约 = 恢复默认。前端可以加个"恢复默认"按钮，点一下就 DELETE。
>
> **小花荣**：那如果预约的铃声和课表默认是同一个呢？
>
> **陈工**：系统比较一下——如果 override.bell_id 等于 task 默认 bell_type 查出来的 bell_id，就不需要任何变更，直接走默认路径。用户设了跟没设一样，没影响。

**张老师总结**：

> 所以这个功能核心改动只有 **5 行代码**，其他全是新增。风险极低，收益却很实在——老师有了一个"不改课表就能换铃"的快捷通道。可以放心做。

### 关键定论

| 问题 | 回答 |
|------|------|
| 改动大吗？ | **很小。唯一改的是 `bell_check_task` 里的 5 行。** 其余全是新增代码 |
| 影响现有系统吗？ | ❌ 完全不影响。无预约时走原路径，性能零损耗 |
| 风险？ | 极低。不碰音频、不碰课表、不碰现有 API |
| 总新增代码 | 约 **220-250 行**（后端 ~90，前端 ~150） |
| 开发时间 | 后端 ~2 小时，前端 ~2-3 小时 |
