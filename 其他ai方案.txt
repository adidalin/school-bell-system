一、 极简版：预约铃声 (Scheduled Overrides)
核心思路：不重构现有的打铃调度，只做最终拦截。

数据层 (models.py)：新增一张表 task_overrides（字段：date, task_id, override_bell_id）。

逻辑层 (engine.py)：在 check_and_ring() 获取到今天需要执行的 task 后，仅仅增加两行代码：去 task_overrides 表查一下，今天这个 task_id 有没有对应的 override_bell_id？如果有，就用新的铃声路径替换原路径，其他逻辑（如区域、时间、节假日判断）一概不动。

展示层 (index.html)：在已有的“排课管理”视图旁边增加一个“临时替换”按钮，点击弹窗选择日期和新铃声即可。不需要开发庞大的日历视图，保持界面的极简和克制。

二、 极简版：TTS 与积木式音频组合 (Sequential Playback)
核心思路：摒弃复杂的底层音频混音（如 FFMPEG 依赖），采用“文件预生成 + 队列顺序播放”模式。

静态化 TTS (app.py)：不要在每次打铃时去请求 TTS API（网络一旦波动，打铃就会卡死或失败）。在前端配置“积木”时，后台调用 Edge TTS 或小米 TTS，将生成的语音直接保存为一个标准的 WAV 文件，存入 static/sounds/ 目录，并在 bells 表中登记为一种特殊类型的铃声（如 type: combo_tts）。

顺序播放 (engine.py)：在 AudioPlayer 类中新增一个 play_sequence(zone_id, filepaths_list) 方法。业务逻辑只需要传入一个列表 ["提示音.wav", "生成的TTS语音.wav", "主音乐.wav"]，引擎依次播放即可。

模块化优势：对系统来说，这三个文件和普通铃声文件没有任何区别，现有的音频驱动、重采样逻辑完全不需要修改，大大降低了维护难度。

三、 极简版：首页实时区域广播 (Instant Broadcast)
核心思路：作为独立的高优先级通道，直接复用 TTS 生成能力。

UI 层 (index.html)：在顶部状态栏下方，用非常克制的高级感 UI 加入一行：[区域多选框] + [文本输入框] + [广播按钮]。这符合现代后台管理面板的直觉设计。

API 层 (app.py)：新增 /api/broadcast。接收到文本后，立刻生成一个临时 WAV 文件（保存在 data/temp/），然后绕过所有课表逻辑，直接调用 engine.manual_bell() 强制播放。播放完毕后自动删除临时文件。

打断逻辑：复用现有的全局停止功能，在下发广播前触发一次区域 stop()，确保不会出现声音重叠。

确实，把“铃声组合”作为第一步是最聪明的决定，它是打通后续所有高级功能的基石。 你想想，一旦把“提示音 + TTS + 音乐”这个顺序播放的底层逻辑跑通了，后面的“首页广播”无非就是一个只含 TTS 的一次性组合，而“预约铃声”无非就是在指定时间播放某个配置好的组合。这完全符合我们确定的“模块化、保持现有结构”的原则。

为了配合 AI 代码助手的开发节奏（尤其是有些工具在每次操作完就自动关掉网页，导致上下文容易断裂），我们需要把这个需求拆解成互不干扰的极简模块，一次只动一层。

以下是针对“铃声组合”的极简落地方案：

一、 数据库的“零侵入”设计
为了不修改现有的排课表和区域绑定逻辑，系统对待“组合铃声”应该就像对待“普通铃声”一样。

不需要新建复杂的关联表。我们直接复用现有的 bells 表：

新增组合时，向 bells 表插入一条记录，bell_type 设为 combo。

把组合的“配方”直接序列化成一段 JSON，存放在原有的 filename 字段里。

例如：{"prefix": "dingdong.wav", "tts": "tts_temp_123.wav", "music": "class_start.wav"}。

优势：现有的课表调度、区域配置完全不需要动任何代码，它们只管读取 bell_id，把压力转移给播放引擎去解析。

二、 播放引擎 (engine.py) 的平滑升级
在 check_and_ring 触发播放时，做一层极简的拦截：

读取到待播放铃声。

判断 bell_type 是否为 combo。

如果是，解析 filename 里的 JSON，把里面的 1~3 个音频文件提取出来，交给一个新增的 play_sequence(files) 方法。

该方法按顺序逐个调用现有的底层音频流播放逻辑，播完一个播下一个。

三、 铃声组合界面的高级感设计
既然要做一个独立的交互界面，我们可以保持极简、高级的专业风格，拒绝传统后台密密麻麻的繁琐表单：

入口：在目前的“铃声文件管理”页面中，增加一个醒目的“➕ 新建组合铃声”按钮。

卡片式拼接 UI：
弹窗内提供三个克制的水平区块（类似积木）：

前置音区块：下拉选择“无”或“叮咚/警报”等预设短音。

TTS 语音区块：一个干净的文本输入框。旁边带一个“生成并试听”的安静按钮。

主铃声区块：下拉选择现有的 MP3/WAV 铃声库。

生成逻辑：点击保存时，前端将文本发给后端 /api/tts/generate，后端调用 Edge TTS 在 static/sounds/ 生成一个实体 WAV 文件返回文件名，最后将这三个区块的文件名打包成 JSON 存入 bells 表。


阶段二：首页实时区域广播 (Instant Broadcast)
这个功能直接站在“铃声组合”的肩膀上。底层其实就是一个“只有 TTS，没有前后音频，且立刻执行”的特殊铃声。

1. 极简 UI 层 (index.html 局部修改)

设计原则：保持界面的克制与专业感。不需要单独开辟大页面。

位置：在“系统总览仪表盘”的最上方，插入一个悬浮质感的操作条。

元素：[多选下拉框：选择广播区域] + [干净的文本输入框：输入要广播的文字] + [主色调按钮：立即广播]。

2. 独立 API 路由 (app.py 追加代码)

新增一个接口 /api/broadcast。

接收前端传来的区域 ID 数组和文本。

复用阶段一写好的 TTS 生成函数，在 static/sounds/ 目录下生成一个形如 broadcast_temp_[时间戳].wav 的临时文件。

3. 引擎强制打断 (engine.py 局部修改)

调用现有的 stop() 方法，让目标区域立刻停止正在播放的常规铃声（避免声音重叠）。

直接调用现有的 play() 播放刚才生成的临时 WAV 文件。播放结束后，开一个异步小线程删掉这个临时文件以节省空间。

阶段三：预约铃声拦截器 (Scheduled Overrides)
这个需求的核心是“时间劫持”。我们不动原有的课表（schedules 和 schedule_tasks）逻辑，只做最终拦截。

1. 轻量级数据表 (models.py 追加)

新增表 scheduled_overrides。

字段极简：id, date (特定日期), time (触发时间, 例: 08:00), zone_id (目标区域), override_bell_id (替换为哪个铃声，这里可以直接选阶段一做好的“组合铃声”)。

2. 引擎执行拦截 (engine.py 核心改造)

在主循环 check_and_ring() 中，当系统根据常规课表算出现在应该播放 A铃声 时，先暂停。

拿当前的 (日期, 时间, 区域) 去 scheduled_overrides 表里查一下。

如果查到了记录，就把 A铃声 替换成 override_bell_id 对应的铃声。如果没查到，继续播放 A铃声。

3. 独立的预约面板 (index.html 追加视图)

新增一个隐藏的 .page 视图，命名为“预约管理”。

界面左侧是“即将生效的预约列表”，右侧是一个干净的表单，用于选择日期、时间、区域和新铃声。

给 AI 助手的开发投喂策略

为了避免 AI 助手产生幻觉或丢失上下文，每次只让它处理一个文件的一个步骤。
例如做阶段一的“铃声组合”时：

第一步：只让它改 app.py，加入 Edge-TTS 的 Python 库调用和 /api/tts/generate 接口。测试能生成音频文件。

第二步：只让它改 engine.py，加入 play_sequence 顺序播放列表的方法。

第三步：只让它改 index.html，把组合铃声的 UI 画出来。

既然整体蓝图已经清晰，我们现在正式动手做第一阶段的“铃声组合”。你是希望先从后端的 app.py 加入 Edge-TTS 接口开始写，还是先从前端 index.html 把高级感的组合 UI 搭出来？
